# Adoption Roadmap — Design

**Date:** 2026-07-03
**Status:** Approved
**Goal:** Grow public adoption of `vite-plugin-shopify-inline-styles`.

## Context

v0.1 is published on npm. The holts theme migration (v0.2) is done, so real
production usage exists and can be measured. The plugin is feature-complete for
its own use case; the adoption bottleneck is trust and visibility, not features.

## Decision

Sequence: **Track A (proof) → Track C (discoverability)**. Track B (flexibility
backlog) stays trigger-driven per the existing ROADMAP principle — external
users generated by A + C are what produce legitimate triggers.

## Track A — Proof

1. **README case study.** Before/after numbers from the holts migration,
   measured with unlighthouse on home, collection, and product pages:
   FCP, LCP, render-blocking request count. One small table plus a sentence
   on methodology (device profile, runs averaged).
2. **`examples/` minimal theme.** One section, one snippet, one
   `vite.config.js` showing `additionalEntrypoints` + `linkEntries`, runnable
   with `npm run build`. Reuse the integration-test fixture where possible
   rather than maintaining a second theme.
3. **CHANGELOG.md.** Keep-a-changelog format, backfilled with v0.1.0.
4. **CI matrix.** Extend the PR workflow to Node 20/22 × Vite 5/6/7
   (peerDep already allows `>=5`). Fix anything the matrix surfaces before
   promoting the plugin.

## Track C — Discoverability (after A)

1. `package.json` keywords (`shopify`, `vite-plugin`, `shopify-theme`,
   `performance`, `inline-css`, `critical-css`) and matching GitHub topics.
2. Comment on barrel/shopify-vite issues/discussions asking about inline CSS,
   linking the plugin as a companion with the case-study numbers.
3. List PRs: awesome-shopify, awesome-vite.
4. Optional: short write-up (Shopify community forum or dev.to) reusing the
   case study.

## Track B — Deferred (unchanged)

Bare path entries, configurable dev delegation target, per-call
`inline: false`, media attribute, non-flat `assetFileNames` warning — all
remain in the ROADMAP backlog with their existing triggers ("someone asks" /
"a real theme hits it"). Not scheduled.

## Non-goals

- No new plugin options or code paths in Track A/C.
- No paid promotion, no starter-theme fork, no standalone (non-companion)
  positioning.

## Risks

- Case-study numbers could be modest. Mitigation: measure first, publish only
  if the story is honest and favorable; either way the measurement informs
  the README's claims.
- Vite 7 matrix may surface incompatibilities. That is the point — fix or
  document before promoting.

## Success signal

External issues/PRs or npm downloads from users outside holts — which then
legitimately fire Track B triggers.
